## 一、概述

### 1.1 项目定位

本项目是一个局域网游戏互联网联机工具——帮助非技术用户将仅支持局域网联机的游戏（如 Minecraft、Terraria 等）通过互联网进行远程联机。核心思路：**将 LAN 游戏协议透明地桥接到远程玩家**。

### 1.2 核心问题

局域网联机游戏默认假设所有玩家在同一广播域内。跨互联网联机需要解决：

1. **网络互通**：一方在 NAT 之后，另一方无法直接连接
2. **延迟敏感**：游戏数据需要低延迟传输
3. **零配置**：用户不应手动配置端口转发或 VPN
4. **协议透明**：不修改游戏本身，对游戏数据完全透明转发

### 1.3 设计原则

- **三路径降级**：IPv6 直连 > P2P TCP 直连 > 中继转发，自动尝试最优路径
- **接口统一**：所有传输方式实现同一 `Transport` 接口，上层无感切换
- **状态驱动**：TunnelManager 基于状态机管理连接生命周期
- **惰性连接**：房主侧游戏服务器断线后不主动重连，等数据到达时再按需重建

## 三、网络检测子系统

### 3.1 检测流程

`NetworkDetector.detect()` 的执行流程：

```
入口
  │
  ├─ 缓存检查（30 秒 TTL）
  │
  ├─ 联网探测（TCP 同时连接 baidu/qq/aliyun，2s 超时）
  │    └─ 无网 → 返回 offline NetworkInfo，5 秒短缓存
  │
  ├─ 并行阶段：
  │    ├─ checkIpv6Capability()
  │    │    ├─ 扫描网卡获取 IPv6 地址
  │    │    ├─ 区分公网地址（排除 fe80::/fc00::/ff00::）
  │    │    └─ TCP 连接 ipv6.google.com:80 验证可达性
  │    │
  │    └─ detectNatType()
  │         ├─ Phase 1: 同一 UDP socket 多点探测映射行为
  │         ├─ 稳定性验证：重查确认映射未改变
  │         └─ Phase 2: CHANGE-REQUEST 探测过滤行为
  │
  └─ 合并为 NetworkInfo，写入缓存
```

### 3.2 NAT 类型检测（RFC 3489/5789 混合实现）

**Phase 1 — 映射行为检测**：

复用一个 UDP socket，顺序向多台 STUN 服务器发送 Binding Request。关键设计：

- **同一 socket → 同一本地端口**。如果 NAT 是 Endpoint-Independent Mapping，所有请求的映射端口应一致；如果是 Address-Dependent Mapping，不同目标 IP 的映射端口不同。
- 收集 3 个响应即提前结束（足够判断）。
- 不足 2 个响应时继续重试剩余服务器，尽力获得第二个数据点。

**稳定性验证****：

重新查询最快的服务器，确认映射地址和端口未发生变化。若发生变化（NAT 映射超时过短或路由器行为异常），保守判定为 Symmetric NAT。

**Phase 2 — 过滤行为检测**：

| CHANGE-REQUEST 参数 | 收到响应 → 判定 |
|---------------------|----------------|
| 换 IP + 换端口 | Endpoint-Independent Filtering（Full Cone） |
| 仅换端口 | Address-Dependent Filtering（Restricted Cone） |
| 均无响应 | Port Restricted Cone（最保守默认） |

至多尝试 2 台延迟最低的服务器，防止长时间等待不支持的服务器。

### 3.3 IPv6 检测

三步骤：
1. `os.networkInterfaces()` 获取所有 IPv6 地址
2. 过滤公网地址（排除 fe80/fc/fd/ff 开头和 ::1）
3. `net.Socket.connect('ipv6.google.com', 80)` 验证可达性

### 3.4 检测结果格式

```typescript
interface NetworkInfo {
  ipv6: {
    available: boolean      // IPv6 外网是否可达
    hasPublicV6: boolean    // 是否有公网 IPv6 地址
    addresses: string[]     // 所有 IPv6 地址
  }
  ipv4: {
    natType: NatType        // full-cone | restricted-cone | port-restricted-cone | symmetric | none | unknown
    publicIp: string        // STUN 发现的公网 IP
    publicPort: number      // STUN 发现的映射端口
    mappingBehavior: MappingBehavior   // endpoint-independent | address-dependent | ...
    filteringBehavior: FilteringBehavior
  }
}
```

### 3.5 设计要点

- **30 秒缓存**：短时间内多次创建/加入房间时避免重复检测
- **离线检测**：无互联网时跳过 STUN 检测直接返回 unknown，5 秒短缓存便于快速重试
- **UDP socket 复用**：整个 NAT 检测周期使用同一个 UDP socket，确保映射一致性

## 四、连接路径选择

### 4.1 路径优先级

| 优先级 | 路径类型 | 条件 | 延迟特征 |
|--------|---------|------|---------|
| 0 | IPv6 直连 | 双方均有公网 IPv6 | <5ms（无 NAT 损耗） |
| 1 | P2P TCP 直连 | 双方 NAT 非 Symmetric | <20ms（NAT 穿透额外 1-2 RTT） |
| 2 | 中继转发 | 始终可用 | >50ms（双程 WebSocket 中转） |

### 4.2 路径选择逻辑

`selectPath(hostNetwork, guestNetwork)` 在 `path-selector.ts` 中实现：

```typescript
function selectPath(host: NetworkInfo, guest: NetworkInfo): ConnectionPath[]

// 决定因素：
// IPv6: hostV6.hasPublicV6 && guestV6.hasPublicV6
// P2P:  !isSymmetric(host.natType) && !isSymmetric(guest.natType)
//         && mappingBehavior !== 'address-and-port-dependent'
// KCP:  仅当双方 mappingBehavior 均为 endpoint-independent
```

### 4.3 P2P 子策略

当 P2P 路径被选中时，`p2pStrategy.methods` 决定尝试顺序：

```
TCP 直连：优先尝试（TCP 更稳定）
UDP KCP：TCP 不可用时尝试（UDP 打洞成功率更高）
```

### 4.4 设计要点

- **保守优先**：Symmetric NAT 下禁止 P2P 尝试，减少 8 秒超时等待
- **unknown + 无公网 IP**：视为本地调试环境（127.0.0.1 环路），允许 P2P
- **Relay 无条件兜底**：确保任何网络环境下至少有一条可用路径

## 五、Transport 接口（核心抽象）

### 5.1 接口定义

所有传输方式统一实现 `Transport` 接口（`connection/types.ts`）：

```typescript
interface Transport {
  readonly type: 'ipv6' | 'p2p' | 'relay'
  readonly status: TransportStatus

  connect(peerInfo: PeerConnectionInfo): Promise<void>
  disconnect(): Promise<void>
  send(data: Buffer): Promise<void>

  // EventEmitter 事件
  on('data', (data: Buffer) => void)
  on('status', (status: TransportStatus) => void)
  on('error', (err: Error) => void)
  on('close', () => void)
  on('traffic', (snapshot: TrafficSnapshot) => void)
}
```

### 5.2 事件说明

| 事件 | 触发时机 | 消费者 |
|------|---------|--------|
| `data` | 收到对端数据 | LocalServer → 写入游戏客户端 / LocalClient → 写入游戏服务器 |
| `status` | 连接状态变更 | TunnelManager 状态机更新 |
| `traffic` | 每秒定时发射 | UI 流量展示 |
| `latency` | WebSocket ping/pong 测量 | UI RTT 展示 |
| `reset` | KCP 带内 RESET 帧 / Relay 重置帧 | LocalClient._handleReset() 重建游戏连接 |

### 5.3 三种实现的差异

| 特性 | Ipv6DirectTransport | P2pTransport | RelayPeerTransport |
|------|--------------------|-------------|-------------------|
| 底层协议 | TCP over IPv6 | TCP over IPv4 | WebSocket + 中继服务器 |
| 角色模型 | active/passive | active/passive | 无角色（共享 WS 连接） |
| 连接建立 | active: TCP connect; passive: 监听 :: | active: 多地址尝试; passive: 监听 0.0.0.0 | 注册事件监听器 |
| 地址交换 | 中继 Signal 传 IPv6 地址和端口 | 中继 Signal 传公网/私网地址 | 无需地址交换 |
| 超时控制 | 5 秒（默认） | 8 秒（可配置） | 无（依赖 WebSocket 超时） |
| 防火墙依赖 | IPv6 端口需放行 | NAT 需支持打洞 | 仅需 HTTPS 出站 |

## 六、隧道机制

### 6.1 房主侧：LocalTunnelClient

```
数据流向：
加入者 ──Transport──→ LocalTunnelClient ──TCP──→ 游戏服务器 (127.0.0.1:gamePort)
                                              （如 Minecraft :25565）
```

核心特性：

- **惰性重连**：游戏服务器断开后不主动重连。Transport 数据到达时才触发 `_triggerLazyConnect()`，连接期间数据缓冲于 `_pendingBuffer`。
- **代际计数器**：`_gen` 字段在每次重连时递增，防止陈旧连接回调覆盖当前连接。
- **待重置标记**：`forceReconnect()` 设置标记，下次 Transport 数据到达时再重建连接，避免立即销毁导致正在写入的数据丢失。
- **重置帧处理**：收到 Transport 的 RESET 事件时，销毁旧 socket、清空缓冲区、立即触发惰性重连。RESET 事件来源：KCP 带内重置信令（0x01 帧）或 Relay 重置帧（payloadLen=0）。

### 6.2 加入者侧：LocalTunnelServer

```
数据流向：
游戏客户端 ──TCP──→ LocalTunnelServer (:localPort) ──Transport──→ 房主
（MC/terraria 等）                          ↑
                                     _writeToAllClients()
                                     ←── Transport data 事件
```

核心特性：

- **多客户端支持**：多个游戏客户端（同游戏的多个玩家）共享同一 Transport 连接。房主返回的数据通过 `_writeToAllClients()` 广播到所有客户端。
- **透明切换 Transport**：`setTransport()` 可随时替换 Transport 实例，用于自动降级场景。旧 Transport 的监听器被 `removeAllListeners` 清除。

### 6.3 数据桥接逻辑

```
加入者游戏客户端             LocalTunnelServer              Transport           LocalTunnelClient          房主游戏服务器
    │                            │                           │                       │                       │
    │   TCP connect              │                           │                       │                       │
    ├───────────────────────────►│                           │                       │                       │
    │                            │  client-connected          │                       │                       │
    │                            ├── emit ──► 重置帧(空Buffer)│                       │                       │
    │                            │                           ├── send(Buffer.alloc(0))►                       │
    │                            │                           │                       ├── _handleReset()      │
    │                            │                           │                       │── destroy old socket   │
    │                            │                           │                       │── triggerLazyConnect() │
    │   send("Hello")            │                           │                       │                       │
    ├───────────────────────────►│  _transport.send(data)     │                       │                       │
    │                            ├──────────────────────────►│  relay/p2p/ipv6       │                       │
    │                            │                           ├──────────────────────►│  _socket.write(data)  │
    │                            │                           │                       ├──────────────────────►│
    │                            │                           │                       │  "Hello"              │
    │                            │                           │                       │◄──────────────────────┤
    │                            │                           │◄──────────────────────┤  _transport.send(echo)│
    │                            │◄──────────────────────────┤  data event           │                       │
    │  _writeToAllClients(data)  │                           │                       │                       │
    │◄───────────────────────────┤                           │                       │                       │
```

## 七、TunnelManager 状态机

### 7.1 状态定义

```
IDLE ──→ CONNECTING_RELAY ──→ CONNECTED (host: 创建房间)
                                │
                                ├──→ CONNECTING_TRANSPORT (guest: 尝试各路径)
                                │       │
                                │       ├──→ CONNECTED (成功)
                                │       └──→ ERROR (所有路径均不可用)
                                │
                                ├──→ DEGRADING (传输断开, 尝试降级)
                                │       │
                                │       ├──→ CONNECTED (降级成功)
                                │       └──→ ERROR (无更多路径)
                                │
                                └──→ DISCONNECTING ──→ IDLE
```

### 7.2 连接流程（加入者侧）

```
joinRoom(roomCode)
    │
    ├─ 1. NetworkDetector.detect()       （~3s，含 30s 缓存）
    │
    ├─ 2. RelayClient.connect()          （~200ms，WebSocket 建连）
    │
    ├─ 3. RelayClient.joinRoom()         （~300ms，中继服务器确认）
    │      └─ 获取 hostNetworkInfo
    │
    ├─ 4. 信号预收集（并行）
    │      ├─ _waitForIpv6Signal()       （1.5s 超时）
    │      ├─ _waitForP2pSignal()        （1.5s 超时）
    │      └─ _waitForKcpSignal()        （1.5s 超时）
    │
    ├─ 5. _selectAndConnect()            （串行尝试各路径）
    │      for path in availablePaths:
    │          await _connectWithPath()   （直到成功或耗尽）
    │
    ├─ 6. _localServer.start()           （绑定本地端口）
    │
    └─ 7. emit('connected')              （通知渲染进程）
```

### 7.3 自动降级

当当前 Transport 的 STATUS 事件为 `error` 时触发 `_degrade()`：

```typescript
async _degrade(): Promise<void> {
  if (this._isDegrading) return   // 防重入
  this._isDegrading = true

  for (let i = nextIndex; i < availablePaths.length; i++) {
    try {
      await this._connectWithPath()  // 创建新 Transport 并建立连接
      return
    } catch (err) {
      logger.warn(`${path} 降级失败`)
    }
  }

  this._setState('error')  // 所有路径均不可用
}
```

**已修复的 Bug**：早期版本使用递归调用导致 `_isDegrading` 死锁。当前版本改用 for 循环，彻底消除重入问题。

## 八、三种传输方式详解

### 8.1 IPv6 直连

**适用场景**：双方均有公网 IPv6 地址（2000::/3 范围）。

**连接建立**：
1. 房主（passive 模式）：`server.listen(bindPort, '::')` — 监听 IPv6 通配地址
2. 加入者（active 模式）：`socket.connect(host, ipv6Address, ipv6Port)`

**地址交换**：房主通过中继 Signal 通道发送 IPv6 地址和监听端口。

**优缺点**：
- ✅ 最低延迟（纯 TCP，无 NAT 损耗）
- ✅ 双栈可同时接受 IPv4 连接
- ❌ 需要双方均有公网 IPv6（国内 IPv6 普及率约 70%）
- ❌ 被动端端口可能被防火墙拦截（需开放或指定固定端口）

**被动端端口安全**：被动端默认使用随机端口（`server.listen(0, '::')`）。可通过 `setBindPort(port)` 指定固定端口以配合防火墙策略。

### 8.2 P2P TCP 直连

**适用场景**：双方 NAT 非 Symmetric，可通过端口预测打洞。

**连接建立**：
1. 房主（passive 模式）：创建临时 TCP 服务器监听 `0.0.0.0:0`（随机端口）
2. 房主通过中继 Signal 将公网 IP:Port 发送给加入者
3. 加入者（active 模式）：优先尝试私有 IP 地址（同 LAN），再尝试公网地址

```typescript
// P2P 地址尝试顺序（active 端）
const targets = [
  ...peerInfo.localAddresses,  // LAN 私有 IP：优先，延迟最低
  peerInfo.publicAddress,       // 公网 IP：NAT 打洞
]

for (const target of targets) {
  await tryConnect(target.host, target.port, 8000ms)
}
```

**优缺点**：
- ✅ 不依赖 IPv6
- ✅ 同 LAN 可走私有 IP，延迟极低
- ❌ Symmetric NAT 100% 失败
- ❌ 端口随机，防火墙需放行出站（仅客户端出站，无需入站规则）

### 8.3 中继转发（Relay）

**适用场景**：无条件兜底，任何网络环境。

**架构**：

```
加入者 ──WebSocket──→ 中继服务器 (ws://host:9800) ←──WebSocket── 房主
                           │
                      [4B sourceIdLen][sourceId][4B payloadLen][payload]
```

**数据帧格式**：

```
房主端解析（_isHost = true）：
  [4B sourceIdLen UInt32BE][N bytes sourceId UTF8][4B payloadLen UInt32BE][N bytes payload]
  payloadLen = 0 → 重置帧（通知房主重建该 Guest 的游戏连接）

加入者端解析（_isHost = false）：
  [4B payloadLen UInt32BE][N bytes payload]
  当 payloadLen 超出缓冲区时，整个消息体作为 payload（兼容中继服务器不发送长度头部的场景）
```

**重置帧协议**：`Buffer.alloc(0)`（payloadLen = 0）通过中继发送时被解释为"游戏客户端重新连接"信号。房主收到后销毁当前游戏 TCP 连接，触发惰性重连，使新客户端的后续数据使用全新连接通道。

**优缺点**：
- ✅ 100% 可用（仅需 HTTPS 出站）
- ❌ 延迟最高（双程 WebSocket 中转 + JSON 序列化/反序列化）
- ❌ 带宽受限（受中继服务器带宽限制）
- ❌ 单点依赖（中继服务器宕机则所有 Relay 连接中断）

### 8.4 三种路径延迟对比

| 路径 | 理论 RTT | 实测 RTT（同运营商） | 备注 |
|------|---------|-------------------|------|
| IPv6 直连 | 0.5-5ms | ~1ms | 纯 TCP，无中间节点 |
| P2P TCP | 1-20ms | ~3ms | NAT 打洞增加 1-2 RTT |
| 中继转发 | 30-100ms | ~45ms | WebSocket + 服务器中转 |

### 9.1 房主创建房间

```
  │

  │

  │
```

### 9.3 离开房间

leaveRoom 的清理顺序（避免资源泄漏）：

1. 断开所有 Guest Transports（房主侧）
2. 断开所有 LocalClients（房主侧）
3. 清理 P2P 超时定时器和备选传输
4. 清理 Relay 备选传输
5. 断开当前 Transport（加入者侧）
6. 停止 LocalServer（加入者侧）
7. 移除事件监听器
8. 离开房间（中继服务器）
9. 断开 Relay WebSocket
10. 重置所有状态变量

### 9.4 重连与恢复

**当前实现**：中继 WebSocket 断开后不自动恢复房间会话。原因：

- 中继服务器在旧连接断开时已清理房间状态
- 重建 WebSocket 无法恢复房间关联
- 用户需手动点击"重新连接"，走完整创建/加入流程

**计划中的改进**（基于 `plan文件`）：RelayClient 缓存房间参数，`reconnectRoom()` 自动重建 WS → 重新创建/加入房间。

## 十、二进制帧协议

### 10.1 中继数据帧

```
Guest → Host 方向（加入者发往房主）：
  [4B payloadLen UInt32BE][N bytes payload]
  示例：0x0000000F "HelloFromGuest!" = 15 字节数据

Host → Guest 方向（房主发往加入者）：
  [4B targetIdLen UInt32BE][M bytes targetId UTF8][4B payloadLen UInt32BE][N bytes payload]
  示例：0x00000009 "member_42" 0x0000000D "EchoFromHost!"

  ── 中继服务器处理后 ──→
  [4B payloadLen UInt32BE][N bytes payload]
  注意：当前中继服务器在 host→guest 方向不包含 [4B payloadLen] 头部，
       客户端已做兼容处理（payloadLen 超界时使用整个消息体）。
```

### 10.2 重置帧

```
payloadLen = 0 → 重置帧
用途：通知房主重建该 Guest 的游戏服务器 TCP 连接
触发：加入者侧 LocalTunnelServer 的 client-connected 事件
```

### 10.3 JSON 控制帧（文本消息）

```typescript
// 通用格式
{ "type": "xxx", "messageId": "req_1_xxx", "data": { ... } }

// 带响应的请求（create-room / join-room）
{ "type": "create-room", "messageId": "req_1_xxx", "data": { gameId, gameName, gamePort, memberName, networkInfo } }
→ 响应：{ "type": "room-created", "messageId": "req_1_xxx", "data": { roomCode, memberId } }

// 无响应的推送（member-joined / member-left / signal）
{ "type": "member-joined", "data": { memberId, memberName, networkInfo } }

// P2P 信令（通过 signal 通道）
{ "type": "signal", "data": { from: "member_xx", signalData: { type: "p2p-address", ip: "...", port: xxxx } } }
```

### 10.4 KCP 帧协议

KcpTransport 在 KCP 数据流上附加 1 字节帧类型前缀，实现数据帧与控制帧的带内复用：

```
帧格式：
  [1B frameType][payload...]

frameType:
  0x00 = DATA  —  应用数据帧。接收方剥除帧头后将 payload 作为 DATA 事件发射
  0x01 = RESET —  控制帧，无 payload。接收方发射 RESET 事件

数据流示例（游戏客户端重连时）：
  [0x00][旧游戏数据...]   ← 残留在 KCP 缓冲区中的旧数据
  [0x01]                   ← RESET 控制帧（通知房主销毁旧 socket）
  [0x00][新游戏数据...]    ← 新游戏会话的数据
```

**设计要点**：

- RESET 帧与 DATA 帧共享同一 KCP 流，KCP 的可靠有序语义保证到达顺序：
  残旧数据 → RESET（销毁旧连接、清空缓冲区）→ 新数据（创建新连接）
- 控制帧仅 1 字节，零额外开销
- 通过 `kcp-transport.ts` 的 `sendControl('reset')` 方法发送，与 `send()` 方法共用同一 KCP 实例
- `_drainKcp()` 按帧类型分发：DATA 帧剥除帧头后压入 `_pendingData` 缓冲区；RESET 帧直接发射 `TRANSPORT_EVENTS.RESET` 事件

**竞态条件消除**：

KCP 带内重置解决了此前 Relay Signal 无法解决的竞态问题：

| 方案 | 信号路径 | 数据路径 | 时序关系 |
|------|---------|---------|---------|
| Relay Signal（旧）| WebSocket (TCP) | KCP (UDP) | TCP 比 UDP 慢，信号晚于游戏数据到达 |
| KCP 带内重置（当前）| KCP 流 (UDP) | KCP 流 (UDP) | 同路径，KCP 保证有序，信号在数据之间按序到达 |

### Bug #1：Relay 降级递归死锁

**问题**：`_degrade()` 在尝试下一路径失败后调用自身（递归），而 `_isDegrading` 标志阻止重复降级。结果：第一次降级尝试失败后，`_isDegrading` 为 true，后续路径永远无法尝试。

**修复**：将递归模式改为 for 循环：

```typescript
// 修复前：_connectWithPath → catch → _degrade() → _connectWithPath → catch → _degrade() → ...
// 修复后：_degrade() → for 循环 → _connectWithPath()（失败 → 继续循环）

for (let i = nextIndex; i < this._availablePaths.length; i++) {
  try {
    this._currentPathIndex = i
    await this._connectWithPath()
    return
  } catch (err) {
    logger.warn(`${path} 降级失败`)
  }
}
```

### Bug #2：Relay Host→Guest 数据帧缺少长度头部

**问题**：中继服务器在 host→guest 方向转发数据时不包含 `[4B payloadLen]` 前缀。加入者侧的 `_handleBinaryData` 将数据前 4 字节错误解析为 payloadLen，导致 payload 前 4 字节被截断。

**现象**：房主发送 "EchoFromHost!"，加入者收到 "FromHost!"（"Echo" 被当作长度头部消耗）。

**修复**：加入者侧增加超界检测：

```typescript
const payloadLen = raw.readUInt32BE(0)
if (BINARY_FRAME_HEADER_SIZE + payloadLen > raw.length) {
  // 中继服务器未添加长度头部，整个消息体即 payload
  payload = raw
} else {
  payload = raw.subarray(BINARY_FRAME_HEADER_SIZE, BINARY_FRAME_HEADER_SIZE + payloadLen)
}
```

### Bug #3：IPv6 TCP 数据竞争

**问题**：`Ipv6DirectTransport` 的 passive 端在 socket `data` 事件发生时，数据监听器可能尚未注册（TunnelManager 在 `connect()` 返回后才注册 `on('data')`）。

**修复**：添加 `_pendingData` 缓冲区 + `drainPendingData()` 回放机制。所有 `data` 事件均入队并发射，延迟注册的监听器可通过 `drainPendingData` 获取已到达的数据。

### Bug #4：心跳超时误判

**问题**：原心跳实现使用固定间隔 `setInterval`，网络抖动时连续丢包即触发误判断连。

**修复**：改用指数退避发送（1s → 2s → 4s → ...，上限 heartbeatInterval）+ 独立 Pong 超时检测。双重条件：距上次心跳成功 > heartbeatTimeout **且** 连续发送失败 >= 3 次才触发重连。

### Bug #5：KCP 内部缓冲区复用导致包头损坏（2026-06-17）

**问题**：KCP UDP 传输在大数据量下连接完全中断。Minecraft 世界数据（~1.5MB）传输时卡在"加入世界中"步骤，下行流量突然停止，约 30 秒后恢复或直接超时。

**根因**：KCP 协议栈的 `flush()` 方法内部使用固定缓冲区 `this.buf` 打包报文段。当缓冲区满或 flush 结束时，output 回调把 `this.buf.slice()` 创建的 Buffer 视图（与 `this.buf` 共享同一底层 `ArrayBuffer`）直接传给 `udpSocket.send()`：

```typescript
// 修复前：this.buf.slice() 是共享视图，send() 异步时可能被覆盖
const kcp = new Kcp(0, 0, (buf: Buffer) => {
  this._udpSocket.send(buf, 0, buf.length, ...)  // buf 是 this.buf 的 slice
})
```

由于 `dgram.Socket.send()` 是异步操作，在 libuv 发出数据前，下一次 `flush()` 循环已经往 `this.buf` 写入了新数据。前一个还在排队发送中的 UDP 数据报内容被覆盖，KCP 头部损坏：

| 字段 | 收到值 | 预期值 | 原因 |
|------|--------|--------|------|
| `conv` | `0x51515151`（KCP_CMD_PUSH 值重复 4 次）| `0` | 覆盖的数据恰好包含 cmd 字节 |
| `token` | 随机垃圾值 | `0` | 被后续 segment 数据覆盖 |
| `len` | 超大的值 | 正常 | 数据区被解释为长度 |

对端 `kcp.input()` 检查 `conv !== this.conv` 后立即返回 `-1`，丢弃所有数据包。多个包被丢弃后 KCP 触发指数退避重传，持续约 30 秒才因 `deadLink` 超时断开。

**触发条件**：任何超过单次 `flush()` 缓冲区容量（1400 字节）的数据发送必然触发。小数据（< 1KB）因单次 flush 即可完成，极少遇到。

**诊断过程**：
1. 单元测试 100KB 传输失败，收不到任何数据
2. 在 `kcp.input()` 添加 debug 日志，发现 `conv` 字段异常（`0x51515151` / `0xA7A7A7A7`）
3. 确认 KCP 实例 `this.conv = 0`（发送时已验证），排除 KCP 状态错误
4. 分析 `flush()` → `output()` → `socket.send()` 链路，定位为 buffer 视图共享导致的数据竞争
5. 单元测试 3 项 1.85s 全部通过，修复确认

**修复**：在 output 回调中用 `Buffer.from(buf)` 创建独立副本再发送：

```typescript
const kcp = new Kcp(0, 0, (buf: Buffer) => {
  if (this._udpSocket && this._peerAddr) {
    const copy = Buffer.from(buf)  // 分配新内存，复制数据
    this._udpSocket.send(copy, 0, copy.length, ...)
  }
})
```

`Buffer.from(buf)` 分配新 `ArrayBuffer` 并复制内容，`socket.send()` 操作的是完全独立的内存，不受后续 `flush()` 写入影响。

**验证**：`tests/unit/tunnel/kcp-transport.test.ts` 覆盖 3 个场景：

| 测试 | 数据量 | 验证点 |
|------|--------|--------|
| 小数据（100KB）| 100KB | 连接可用，数据完整性 |
| MC 世界数据突发 | 1.5MB（53 个分片，47B–21KB）| 大数据完整性，无死锁 |
| 双向交替传输 | 100 条发送 + 回复 | 双向通信不死锁 |

3 项测试全部通过，耗时 ~1.85s。

**经验教训**：Node.js `Buffer.slice()` 返回共享内存的视图而非副本。传递给异步 API（`socket.send()`、`fs.write()` 等）时，若原始 buffer 后续被修改，会导致数据损坏。向异步 API 传递 Buffer 时，如不能保证其生命周期，应主动 `Buffer.from(src)` 复制。

### Bug #6：游戏退出重连竞态条件（2026-06-17）

**问题**：玩家退出游戏后重新打开，KCP 传输持续产生残留数据导致新游戏会话无法正常建立。Minecraft 服务器列表无法 ping 通，也无法再次加入游戏，直到数据停止交互后才能恢复正常。

**根因**：游戏退出后，KCP 流中仍残留旧的游戏服务器数据（处于 KCP 内部发送/接收缓冲区）。当游戏重新连接时，新数据与残留旧数据混合到达房主侧，导致协议协商混乱（Minecraft STATUS/LOGIN 协议状态错乱）。

更根本的原因是多路径竞态：重置信令通过 Relay Signal（WebSocket TCP）发送，而游戏数据通过 KCP（UDP）传输。UDP 比 TCP 快，新游戏数据在重置信令到达房主之前就已到达，旧 socket 仍处于活跃状态，新旧数据混合写入游戏服务器。

**修复方案**：KCP 带内重置（In-band Reset）。

1. KCP 数据流引入 1 字节帧类型前缀（0x00=DATA, 0x01=RESET）
2. `KcpTransport.send()` 自动为应用数据添加 0x00 帧头
3. 新增 `KcpTransport.sendControl('reset')` 发送 0x01 帧，与数据共享同一 KCP 流
4. `KcpTransport._drainKcp()` 解析帧类型，RESET 帧发射 `TRANSPORT_EVENTS.RESET` 事件
5. 加入者侧游戏客户端重连时，优先通过 KCP 发送 RESET 帧，再发送游戏数据

**时序**：

```
游戏退出                   游戏重连
  │                          │
  ▼                          ▼
┌──── KCP 流 ────────────────────────────────────►
│ [0x00][旧数据...][0x00][旧数据][0x01][0x00][新数据][0x00][新数据]...
│                              ▲      ▲            ▲
│                              │      │            └── 新游戏会话数据（缓冲 → 惰性重连后写入）
│                              │      └── RESET 帧: _handleReset() 销毁旧 socket, 清空缓冲区
│                              └── KCP 保证残留旧数据先到达，RESET 在其后
```

**受影响文件**：
- `src/core/tunnel/kcp-transport.ts` — 添加帧协议和 `sendControl()` 方法
- `src/core/tunnel/tunnel-manager.ts` — `client-reconnected` 处理器优先使用 KCP 带内重置
- `src/core/tunnel/local-client.ts` — 已有 `_handleReset()` 方法，未修改

**验证方式**：端到端测试，退出 Minecraft 后重新打开，验证服务器列表可正常 ping 通且可进入游戏，日志中无协议错误。重复 5 次均正常则确认修复。

## 十二、局限性与优化方向

### 12.1 已知局限

| 局限 | 影响 | 可能方案 |
|------|------|---------|
| Symmetric NAT 无法 P2P | 约 10-20% 国内 NAT 为 Symmetric | 引入端口范围扫描（如 frp） |
| 中继服务器单点 | 所有 Relay 流量经过一台服务器 | 中继集群 + 自动故障切换 |
| 无连接池 | 每个 Guest 独立 TCP 连接 | 游戏协议不适用（按连接区分玩家） |
| 无会话恢复 | 中继断线后房间失效，需手动重连 | `reconnectRoom()` 计划中 |
| P2P 无 ICE | 使用简化的地址交换而非完整 WebRTC | 引入 WebRTC 的 ICE 流程 |
| IPv6 passive 端口随机 | 防火墙需开全部高端口 | `setBindPort()` 已支持固定端口 |
| 未加密传输 | 中继数据以明文转发 | 计划加入 TLS + 端到端加密 |

### 12.2 推荐优化方向

**短期**：
- 实现 `reconnectRoom()`（计划已制定），支持中继断线后自动恢复
- 引入 WebRTC ICE + DataChannel 替代 TCP P2P，提高 NAT 穿透率
- 中继服务器部署集群 + Nginx 反代 WebSocket 负载均衡

**中期**：
- 端到端 AES 加密中继数据
- 加入带宽估算和自适应流控
- 支持中继服务器动态切换（监测延迟，自动切换最优服务器）

**长期**：
- P2P 连接多路复用（共享同一 UDP 端口管理多个对等连接）
- IPv6 端口预测（学习操作系统临时端口分配规律，提前通知防火墙放行）
- 分布式中继网络（参考 Tor 的 Onion Routing 思路）
