## 1. 项目概述

### 1.1 目标
开发一款桌面应用，帮助非技术用户将**单机局域网联机游戏**（如 Minecraft、Terraria 等）通过互联网进行联机。用户无需了解端口转发、路由器配置、防火墙等网络知识，即可一键创建或加入远程游戏房间。

### 1.2 核心场景
- **房主（Host）**：启动游戏后打开本软件，软件自动检测到本机运行的游戏服务器 → 一键创建房间 → 生成房间码/链接分享给好友
- **加入者（Join）**：输入好友的房间码 → 软件自动建立隧道连接 → 加入者启动游戏 → 在局域网列表中看到房主的游戏 → 点击加入

### 1.3 支持平台
| 平台 | 支持等级 |
|------|---------|
| Windows 10/11 | 原生支持 |
| macOS 11+ (Intel / Apple Silicon) | 原生支持 |
| Linux (Ubuntu 20.04+, 主要发行版) | 原生支持 |

**关键决策：核心引擎运行在主进程**。原因：
1. 核心引擎需要访问 Node.js 原生网络 API（`dgram`、`net`），这些在渲染进程中不可用（即使启用 `nodeIntegration` 也不推荐）
2. 渲染进程崩溃不应中断正在进行的隧道连接
3. 系统托盘可以在窗口关闭后继续运行

### 4.3 数据流

```
【房主创建房间】
  1. Renderer: 点击"创建房间"
  2. IPC:      create-room
  3. Core:     GameDetect 查找本机游戏进程 → Discovery 确认端口
  4. Core:     NetworkDetect 检测本机网络（IPv6 能力、NAT 类型）
  5. Core:     Tunnel 建立本地隧道服务端（监听 IPv4 + IPv6）
  6. Core:     RelayClient 上报房间信息 + 房主网络信息到 Relay Server
  7. IPC:      返回 房间码 + 房主网络状态
  8. Renderer: 显示房间码 + 网络状态 + 连接状态

【加入者加入房间】
  1. Renderer: 输入房间码 → 点击"加入"
  2. IPC:      join-room
  3. Core:     NetworkDetect 检测本机网络（先联网探测 → IPv6 + NAT 并行）
  4. Core:     RelayClient 从 Relay Server 获取房主网络信息
  5. Core:     条件性等待房主信号（仅等待本机支持的路径信号，1500ms 超时）
  6. Core:     路径选择 + 按序尝试：
                ┌─ 双方 IPv6 → Ipv6DirectTransport
                ├─ 仅 IPv4 + 可 P2P → P2pTransport
                └─ 兜底 → RelayPeerTransport
  7. Core:     TunnelManager 建立本地隧道 + 选定传输
  8. Core:     本地监听 127.0.0.1:虚拟端口
  9. IPC:      返回 虚拟端口号 + 连接方式
 10. Renderer: 提示"已通过 XXX 方式连接，请连接 127.0.0.1:端口"
 11. User:     在游戏中添加局域网游戏 → 输入 127.0.0.1:端口
```

## 5. 核心功能设计

### 5.1 LAN 游戏发现 (`src/core/discovery/`)

#### 5.1.1 UDP 广播扫描

```
Host 机器：
  → 广播 UDP 包到 255.255.255.255:发现端口
  → 包含本机信息（游戏名称、端口、版本）
  → 等待响应

Scanner 机器：
  → 监听 UDP 发现端口
  → 收到广播后解析游戏信息
  → 通过 IPC 通知渲染进程更新列表
```

- 扫描端口范围：可配置，默认扫描常见游戏端口
- 扫描间隔：每 3 秒广播一次，持续 30 秒
- 游戏退出后自动从列表移除（超时 15 秒无响应）

#### 5.1.2 已知游戏数据库 (`game-db.ts`)

```typescript
interface GameEntry {

  detectPorts: number[]; // 常见端口
  processNames: string[]; // 进程名（用于本地检测）

```

### 5.2 网络检测 (`src/core/network-detect/`)

#### 5.2.1 概述

每次加入者连接房间前，双方独立进行网络环境检测。房主在网络检测后将其结果上传到 Relay Server，加入者获取房主信息后与本机结果对比，选择最优连接路径。

检测模块由四部分组成：

1. **联网探测** — 快速 TCP 连接测试，判断本机是否已联网（无网络时跳过后续检测）
2. **IPv6 能力检测** — 获取本机 IPv6 地址列表、判断公网 IPv6 可达性
3. **NAT 类型检测** — 基于 STUN 协议，两阶段判定 NAT 的映射行为和过滤行为
4. **检测编排器** — 先执行联网探测，再并行执行 IPv6 + NAT 检测，合并结果、管理缓存

#### 5.2.2 检测内容

```typescript
interface NetworkInfo {
  ipv6: {
    available: boolean           // 是否有全局单播 IPv6 地址
    hasPublicV6: boolean         // 是否具有可达的公网 IPv6
    addresses: string[]          // 本机 IPv6 地址列表（排除链路本地 fe80::）
  }
  ipv4: {
    natType: NatType             // 经典 NAT 类型（none / full-cone / restricted-cone / port-restricted-cone / symmetric / unknown）
    publicIp: string             // 公网 IPv4 地址（通过 STUN 获取）
    publicPort: number           // 公网端口映射
    mappingBehavior: MappingBehavior   // RFC 5780 映射行为
    filteringBehavior: FilteringBehavior  // RFC 5780 过滤行为
  }
}
```

**NAT 类型（经典分类）：**

| 类型 | 说明 | P2P 可能性 |
|------|------|-----------|
| `none` | 无 NAT（公网 IP） | ✅ 最高 |
| `full-cone` | 完全锥型 NAT | ✅ 高 |
| `restricted-cone` | IP 限制锥型 NAT | ✅ 中 |
| `port-restricted-cone` | 端口限制锥型 NAT | ✅ 低 |
| `symmetric` | 对称型 NAT | ❌ 不可 |
| `unknown` | 无法检测 | ❌ 不确定 |

#### 5.2.3 检测流程（两阶段 + 稳定性验证）

全程使用**同一 UDP socket**，确保 NAT 映射一致性：

```
detectNatType():
  │
  ├── socket 创建（绑定随机端口）
  │
  ├── Phase 1: STUN Binding Request（多服务器映射行为检测）
  │     │
  │     向 5 台 STUN 服务器发送 Binding Request（顺序，早停优化）
  │     收集到 3 个响应即提前退出（足够判断映射行为）
  │     不足 2 个时尽力尝试剩余服务器，凑足第二个数据点
  │     │
  │     ├── 全部失败 ──▶ unknown（返回）
  │     │
  │     ├── 公网 = 本机内网地址 ──▶ none（返回）
  │     │
  │     └── 映射稳定性验证
  │           重新查询最快服务器，确认映射地址/端口未变化
  │           ├── 映射变化 ──▶ 保守判为 Symmetric（返回）
  │           └── 映射稳定 ──▶ 进入 Phase 2
  │
  ├── 映射行为判定
  │     ├── ≥ 2 台不同端口 ──▶ Address-Dependent Mapping = Symmetric（返回）
  │     └── ≥ 2 台同端口 / 1 台已验证 ──▶ Endpoint-Independent Mapping
  │
  └── Phase 2: CHANGE-REQUEST（过滤行为检测）
        │
        至多尝试 2 台延迟最低的服务器（RFC 3489 CHANGE-REQUEST 属性）
        │
        ├── 换 IP+端口能收到 ──▶ Endpoint-Independent Filtering = Full Cone
        ├── 仅换端口能收到   ──▶ Address-Dependent Filtering = Restricted Cone
        └── 均无响应         ──▶ Port Restricted Cone（最保守默认）
```

**关键设计要点：**

| 设计决策 | 理由 |
|---------|------|
| 同一 socket 全程复用 | 保持同一 NAT 映射，不同 socket 必然产生不同映射端口，造成 Symmetric 假阳性 |
| Phase 1 收集 3 响应即早停 | 映射行为只需 2 个不同目标即可判定，3 个足够排除巧合；早停降低整体检测时间 |
| 映射稳定性验证 | 捕获 NAT 映射超时过短或路由器行为异常的情况，防止伪 Endpoint-Independent |
| Phase 2 限 2 台服务器 | CHANGE-REQUEST 属性在 RFC 5389 中已废弃，多数现代服务器不支持；试 2 台足够 |
| 每轮 1.5s 超时 + 2 次重试 | UDP 丢包场景下保证可靠性，同时避免长时间阻塞 |

#### 5.2.4 RFC 5780 行为类型

**映射行为（Mapping Behavior）** — 决定 P2P 打洞策略：

| 行为类型 | 含义 | P2P 影响 |
|---------|------|---------|
| `endpoint-independent` | 对任意目的地址复用同一映射端口 | ✅ 单次 STUN 获取的映射地址对所有 peer 有效 |
| `address-dependent` | 不同目的 IP 使用不同映射端口 | ⚠️ 不同 peer 看到不同映射端口，需要为每个 peer 单独探测 |
| `address-and-port-dependent` | 不同目的 IP:端口 使用不同映射端口 | ❌ 等价于 Symmetric NAT，UDP 打洞几乎必然失败 |
| `unknown` | 无法检测 | ❌ 保守阻断 |

**过滤行为（Filtering Behavior）** — 决定打洞的握手方式：

| 行为类型 | 含义 | P2P 影响 |
|---------|------|---------|
| `endpoint-independent` | 任意外部主机均可向映射地址发包 | ✅ 内网发一个包后，任意外部地址都能打回来 |
| `address-dependent` | 仅本机曾联系过的 IP 可发回 | ⚠️ 需要先让 peer 从正确 IP 发起 |
| `address-and-port-dependent` | 仅本机曾联系过的精确 IP:端口 可发回 | ❌ 打洞窗口极窄 |
| `unknown` | 无法检测 | ❌ 保守默认 |

**行为类型到经典类型的映射关系：**

| 映射行为 | 过滤行为 | 经典 NAT 类型 |
|---------|---------|-------------|
| endpoint-independent | endpoint-independent | Full Cone |
| endpoint-independent | address-dependent | Restricted Cone |
| endpoint-independent | address-and-port-dependent | Port Restricted Cone |
| address-dependent | address-and-port-dependent | Symmetric |

#### 5.2.5 鲁棒性措施

| 场景 | 应对措施 |
|------|---------|
| UDP 丢包 | 每次查询自动重试 2 次，丢弃不匹配 Transaction ID 的响应 |
| 仅单服务器响应 | 尽力重试剩余服务器获取第二数据点；单点通过稳定性验证做最佳推测 |
| STUN 服务器全部不可达 | 返回 `unknown`，路径选择器绕过 P2P 直连，直接使用 Relay |
| NAT 映射超时过短 | 稳定性验证（Phase 1 后重新查询最快服务器），映射变化则判为 Symmetric |
| 服务器不支持 RFC 3489 | Phase 2 限 2 台并设置短超时，均失败时默认 Port Restricted Cone（最保守锥型）|
| 检测时间过长 | Phase 1 早停（3 响应退出）、Phase 2 限 2 台、每轮 1.5s 超时封顶 |

- NAT 检测使用 RFC 3489 STUN CHANGE-REQUEST 机制区分三种锥型 NAT，退化时默认最保守类型
- IPv6 检测使用操作系统网络接口 API + 实际 TCP 监听验证
- 两项检测并行执行，典型耗时约 1-3 秒

#### 5.2.6 联网探测（Connectivity Check）

检测开始时先执行快速的联网探测，利用 TCP 连接测试判断本机是否已联网：

```typescript

#### 5.2.7 双 TTL 缓存策略

检测结果使用模块级缓存，根据结果状态使用不同的过期时间：

| 场景 | 缓存 TTL | 理由 |
|------|---------|------|
| 在线（检测成功） | 30 秒 | 网络环境短期稳定，减少重复检测 |

### 5.3 连接路径选择 (`src/core/connection/`)

#### 5.3.1 路径选择算法

```
PathSelector.select(hostInfo: NetworkInfo, guestInfo: NetworkInfo): ConnectionPath

  优先级 1 ── IPv6 直连
    条件：hostInfo.ipv6.hasPublicV6 === true
          guestInfo.ipv6.hasPublicV6 === true
    方式：TCP Socket 直连（net.createConnection）
    优点：零 NAT 问题、无中继成本、最低延迟、最高吞吐
    └── 失败 ──▶ 降级到优先级 2

  优先级 2 ── IPv4 P2P（基于 RFC 5780 MappingBehavior 判定）
    条件：双方 NatType 均非 symmetric/unknown
         + 双方 mappingBehavior 均非 address-and-port-dependent/unknown
    方式：TCP 直连（非 WebRTC），携带 p2pStrategy 供传输层参考
    优点：无中继成本、延迟低
    └── 失败 ──▶ 降级到优先级 3

  优先级 3 ── Relay 中继
    条件：无条件（兜底方案）
    方式：双方连接到 Relay Server，通过 WebSocket 二进制帧转发
    优点：100% 可用（依赖服务器）
```

**P2P 判定逻辑（两层级联）：**

```
第一阶段（NatType 基准阻断）：
  symmetric / unknown ──▶ 直接阻断，不尝试 P2P

第二阶段（MappingBehavior 精细化判定）：
  address-and-port-dependent ──▶ 阻断（等价 Symmetric NAT）
  unknown ──▶ 阻断（无法确定）
  endpoint-independent / address-dependent ──▶ 允许尝试 P2P
```

**P2P 策略信息（p2pStrategy）：**

路径选择器在返回 P2P 路径时附带双方的 NAT 行为类型：
```typescript
{
  type: 'p2p',
  priority: 2,
  description: 'P2P 直连',
  p2pStrategy: {
    host: { mappingBehavior: 'endpoint-independent', filteringBehavior: 'address-dependent' },
    guest: { mappingBehavior: 'endpoint-independent', filteringBehavior: 'address-and-port-dependent' }
  }
}
```

`selectPath()` 返回最优单路径，`getSortedPaths()` 返回完整的有序候选列表供逐级尝试：

`P2pTransport` 可据此决定打洞方式：
- `endpoint-independent` 双方 → 标准 UDP 打洞，无需额外探测
- `address-dependent` 存在 → 启用 ICE Peer Reflexive 机制，允许映射端口变化时自适应
- `address-and-port-dependent` 存在 → 不走 P2P（已由 PathSelector 阻断）

#### 5.3.2 每加入者独立选路

```
           ┌─────────── 加入者 A (有 IPv6) ───▶ IPv6 TCP 直连
           │
房主 ──────┼─────────── 加入者 B (无 IPv6, FullCone NAT) ───▶ IPv4 P2P
           │
           └─────────── 加入者 C (无 IPv6, Symmetric NAT) ───▶ Relay 中继

每个加入者独立走自己的最优路径，互不影响。
```

- 房主持有所有已建立连接的引用，流量分别转发到对应的游戏端口
- 加入者 A 的高吞吐 IPv6 直连不影响加入者 C 的中继连接
- 中继服务器仅服务无法 P2P 的加入者，大幅降低服务器负载

#### 5.3.3 动态切换

如果某一加入者的连接方式中断，自动降级到下一优先级：

```
IPv6 直连断开 ──▶ 尝试 P2P ──▶ 失败则走 Relay（保持连接不断）
```

切换过程对上层游戏透明（游戏 TCP 连接无需重连，新隧道连接后继续转发）。

### 5.4 TCP 隧道 (`src/core/tunnel/`)

#### 5.4.1 隧道架构

IPv6 直连模式：

P2P 模式：
┌─────────┐      ┌──────────┐   P2pTransport(TCP)    ┌──────────┐      ┌─────────┐

                                   └──────────────┘
```

#### 5.4.2 连接策略总览

| 优先级 | 连接方式 | Transport 实现 | 条件 | 优势 | 劣势 |
|--------|---------|---------------|------|------|------|
| 1 (最优) | IPv6 TCP 直连 | `Ipv6DirectTransport` | 双方均有全局 IPv6 | 原生 TCP、最低延迟、零中继成本 | IPv6 尚未全面普及 |
| 2 | IPv4 P2P | `P2pTransport` (TCP 直连) | 双方 NAT 可穿透（NatType + MappingBehavior 二层判定） | 低延迟、无中继成本 | 对称 NAT / address-and-port-dependent 无法穿透 |
| 3 (兜底) | Relay 中继 | `RelayPeerTransport` (WebSocket 二进制帧) | 无条件 | 100% 可达 | 依赖服务器、延迟增加 |

#### 5.4.5 Relay 中继转发机制

##### 概述

Relay 中继是连接路径选择中的兜底方案（优先级 3），当 IPv6 直连和 P2P 均不可用时自动降级启用。数据通过 Relay Client 的 WebSocket 连接经中继服务器转发，对上层 TunnelManager 和游戏完全透明。

整个转发链路：

```
游戏服务器 (:25565)
    ↕ TCP (本机)
LocalTunnelClient (房主侧)
    ↕ Transport 接口
RelayPeerTransport (房主侧)
    ↕ WebSocket 二进制帧 (WSS)
中继服务器
    ↕ WebSocket 二进制帧 (WSS)
RelayPeerTransport (加入者侧)
    ↕ Transport 接口
LocalTunnelServer (加入者侧)
    ↕ TCP (本机)
游戏客户端 (:25565)
```

##### 二进制帧协议

TCP 数据通过 WebSocket 二进制帧传输，采用长度前缀（Length-Prefixed）格式：

```
二进制帧格式：
┌────────────────────────────────────────────────────────────┐
│  [4B Big-Endian — 负载长度]  │        [负载数据]            │
│                                                             │
│  payloadLen = 0  ──▶  重置帧（控制信号，无负载）              │
│  payloadLen > 0  ──▶  普通数据帧，长度由 payloadLen 指定      │
└────────────────────────────────────────────────────────────┘

编码示例：
  const header = Buffer.alloc(4)
  header.writeUInt32BE(payload.length, 0)
  const frame = Buffer.concat([header, payload])
  ws.send(frame)

解码逻辑：
  const payloadLen = innerData.readUInt32BE(0)
  if (payloadLen === 0) {
    // 重置帧 — 无负载，仅作为带内控制信号
    return
  }
  // 普通数据帧
  const actualData = innerData.subarray(4, 4 + payloadLen)
```

**设计要点**：
- 大端序（Big-Endian）UInt32，支持最大 4GB 单帧
- 控制消息（JSON 文本帧）与数据帧（二进制帧）在同一 WebSocket 连接上共存，通过帧类型（opcode）区分
- JSON 控制帧用于房间管理、信令中继、心跳，二进制帧用于游戏数据

##### 重置帧机制（Reset Frame）

`Buffer.alloc(0)` 通过 `Transport.send()` 发送时触发重置帧机制：

```
触发场景：
  加入者侧本地隧道服务端收到新的游戏客户端连接
    → 发送 Buffer.alloc(0) 到 Transport
    → 中继服务器检测到 payloadLen === 0
    → 向房主侧转发 reset 事件
    → RelayPeerTransport 发射 TRANSPORT_EVENTS.RESET
    → LocalTunnelClient._handleReset():
        1. 销毁当前游戏服务器 socket
        2. 清空待发送缓冲区
        3. 递增代际计数器 (_gen++)
        4. 触发惰性重连

重连路径：
  LocalTunnelClient
    → _triggerLazyConnect()
    → _doConnect() (递增 _gen，创建新 TCP socket)
    → 连接到游戏服务器 :port
    → 刷新缓冲区 (_flushBuffer)
```

**设计动机** — 适配 Minecraft 多连接单协议模式：

Minecraft 协议中，客户端对服务器的连接分为两个阶段：

| 阶段 | 握手状态 | 说明 |
|------|---------|------|
| STATUS (服务器列表 Ping) | state=1 | 查询 MOTD、在线人数、协议版本 |
| LOGIN (实际加入游戏) | state=2 | 登录认证、进入游戏 |

这两个阶段由**独立的 TCP 连接**完成。游戏启动时先发 STATUS 请求获取服务器列表，玩家点击加入时才发起 LOGIN 连接。两次连接是独立的 TCP 流。

**问题**：隧道建立后，首次 STATUS 请求通过隧道到达游戏服务器后，房主侧的 `LocalTunnelClient` 连接到游戏服务器的 TCP socket 处于 ESTABLISHED 状态。当玩家点击加入游戏时，新连接（LOGIN）到达加入者侧的 `LocalTunnelServer`，但房主侧的 socket 仍是旧连接，游戏服务器会在旧 socket 上发送 LOGIN 阶段的响应，导致数据错乱。

**解决方案**：每次有新的游戏客户端连接加入者本地隧道时，发送一个重置帧通知房主侧断开旧连接并重新连接游戏服务器。这样 LOGIN 阶段的数据会通过新 socket 发送，实现正确的协议隔离。

```
时间线：
  隧道建立完成
    │
  ── 玩家启动游戏 ──────────────────────────
    │
  ① 游戏发送 STATUS 握手 (state=1) 到 127.0.0.1:虚拟端口
  ② LocalTunnelServer 接收连接，数据通过 Transport 发送到房主
  ③ 房主 LocalTunnelClient 连接游戏服务器 :25565
  ④ STATUS 请求通过该 socket 到达游戏服务器
  ⑤ 游戏服务器返回服务器列表信息
    │
  ── 玩家点击"加入服务器" ──────────────────
    │
  ① 游戏发送 LOGIN 握手 (state=2) 到 127.0.0.1:虚拟端口
  ② LocalTunnelServer 检测到新客户端连接
  ③ 发送重置帧 (Buffer.alloc(0)) 到 Transport
  ④ 房主侧 LocalTunnelClient 收到重置：
      - 销毁旧 socket（STATUS 阶段的连接）
      - 创建新 socket 到游戏服务器
  ⑤ LOGIN 握手数据通过新 socket 发送
  ⑥ 游戏服务器正确识别 LOGIN 握手
```

##### 每加入者独立传输隔离

房主侧为每个加入者维护独立的 Relay 传输通道：

```
房主进程内：
  ┌──────────────────────────────────────────────────────┐
  │ TunnelManager                                         │
  │                                                       │
  │  _guestTransports: Map<string, Transport>              │
  │    ├── "member-A" → RelayPeerTransport(target=A)       │
  │    │                └── LocalTunnelClient → GameServer │
  │    │                                                  │
  │    ├── "member-B" → RelayPeerTransport(target=B)       │
  │    │                └── LocalTunnelClient → GameServer │
  │    │                                                  │
  │    └── "member-C" → RelayPeerTransport(target=C)       │
  │                     └── LocalTunnelClient → GameServer │
  │                                                       │
  │  _relayClient: RelayClient (单一 WebSocket 连接)       │
  └──────────────────────────────────────────────────────┘
```

每个 `RelayPeerTransport` 通过 `_targetMemberId` 过滤：
- **接收过滤**：`_boundOnData` 检查 `sourceMemberId` 是否匹配目标成员，不匹配则丢弃
- **发送定位**：`send()` 调用 `relayClient.sendData(data, this._targetMemberId)`，中继服务器据此将数据路由到对应加入者
- **重置帧过滤**：`_boundOnReset` 同样按 `_targetMemberId` 过滤

这种隔离设计确保：
- 加入者 A 的流量不会泄漏到加入者 B
- 每个加入者可独立使用不同的传输方式（IPv6 / P2P / Relay）
- 一个加入者断开不影响其他加入者的连接
- 重置帧仅触发对应通道的游戏连接重建

##### 中继断线恢复

`RelayPeerTransport` 注册 `RelayClient` 的 `disconnected` / `connected` 事件：

```
WebSocket 断连
    │
  RelayClient 发射 'disconnected'
    → RelayPeerTransport 更新状态为 'disconnected'
    → TunnelManager 发射 TRANSPORT_EVENTS.STATUS
    → TunnelManager._setupTransportEvents 检查路径：
        房主侧：等待加入者重新连接，不做路径切换
        加入者侧：触发自动降级到下一优先级路径
    │
  RelayClient 指数退避重连（最多 5 次）
    │
  重连成功
    RelayClient 发射 'connected'
    → RelayPeerTransport 更新状态为 'connected'
    → 传输自动恢复
```

- 房主侧的中继断开不影响已建立的游戏连接，等待加入者自动重连
- 加入者侧的中继断开触发自动降级（如果还有其他可用路径），否则持续等待重连

##### 与惰性重连的协同

加入者侧的 `LocalTunnelClient` 使用惰性重连策略：

- 不主动维护到游戏服务器的 TCP 连接
- 仅当 Transport 有数据到达时才尝试连接游戏服务器
- 连接建立期间数据暂存于 `_pendingBuffer`，连接成功后按序刷新

中继场景下的数据流：

```
中继数据到达 (RelayPeerTransport.emit DATA)
    │
  LocalTunnelClient.setTransport DATA handler
    ├── socket 已就绪 → 直接 socket.write(data)
    └── socket 未就绪 → 追加到 _pendingBuffer
                         → _triggerLazyConnect()
                            → _doConnect()
                               → TCP 连接到游戏服务器
                               → 连接成功 → _flushBuffer()
                                  → 按序写入所有积压数据
```

这种设计避免无效连接：如果无流量经过中继（例如加入者已启动游戏但尚未操作），`LocalTunnelClient` 不会浪费资源维护到游戏服务器的空闲 TCP 连接。

##### 关键设计决策总结

| 决策 | 选择 | 理由 |
|------|------|------|
| 传输协议 | WebSocket (WSS) | 浏览器兼容、NAT 友好、TLS 加密 |
| 帧格式 | 4 字节大端长度前缀 | 简单可靠、无歧义的成帧协议 |
| 控制/数据复用 | 同连接不同 opcode | 避免额外的连接管理，简化 NAT 映射 |
| 每加入者隔离 | 独立 RelayPeerTransport | 流量隔离、独立降级、独立重置 |
| 重置帧编码 | payloadLen=0 的空帧 | 无需额外消息类型，利用既有成帧协议 |
| 游戏连接策略 | 惰性重连 | 避免空闲连接浪费资源 |
| 房主/加入者角色 | 通过 targetMemberId 区分 | 中继服务器无状态转发，client 端处理路由 |

#### 5.4.6 P2P TCP 直连方案（非 WebRTC）

使用原生 TCP 直连而非 WebRTC DataChannel，避免 `node-datachannel` 原生编译依赖：

- **信令**：通过 Relay Client 的 WebSocket 交换连接信息（IP:Port）
- **连接**：Active 方尝试 TCP 直连 Passive 方暴露的地址
- **同 LAN 检测**：优先使用私有 IP 地址（192.168.x.x / 10.x.x.x），避免绕行 NAT
- **NAT 穿透**：利用 STUN 发现的公网 IP:Port 尝试 TCP 打孔
- 仅当双方 IPv6 检测失败且 NAT 类型非 Symmetric/Unknown 时才启用

#### 5.4.7 TCP ↔ WebSocket 桥接 (`relay-peer.ts`)

RelayPeerTransport 实现 Transport 接口，通过 WebSocket 二进制帧在中继服务器和本地隧道之间桥接 TCP 数据：

```
本地 TCP Socket ──► RelayPeerTransport.send() ──► WebSocket 二进制帧 ──► 中继服务器 ──► 远端 WebSocket ──► RelayPeerTransport.emit('data') ──► 本地隧道
```

每个加入者对应独立的 `RelayPeerTransport` 实例，通过 `_targetMemberId` 过滤接收数据。

#### 5.4.8 IPv6 直连通道 (`ipv6-direct.ts`)

`Ipv6DirectTransport` 实现 Transport 接口，利用 IPv6 公网地址直接互联：

- **Active 模式**（加入者侧）：`connect(remoteAddr)` — 通过 `net.createConnection` 连接到房主的 IPv6 地址
- **Passive 模式**（房主侧）：`connect()` — 在本地监听随机端口，`listening` 事件触发后立即 resolve，通过 Relay Signal 将监听地址通知加入者
- 双方均有公网 IPv6 时优先使用，零中继成本

**Passive 模式关键设计**：`connect()` 在 `listening` 事件触发时 resolve，而非等待 TCP 连接到达。这样房主可以立即通过 Signal 将监听地址发送给加入者，避免死锁：

```typescript
server.on('listening', () => {
  this._localPort = addr.port
  resolve() // 立即 resolve，不等 TCP 连接
})
```

#### 5.4.9 TunnelManager 隧道生命周期管理 (`tunnel-manager.ts`)

TunnelManager 是隧道建立的核心编排器，负责根据网络检测结果创建对应传输、管理降级和中继清理。

##### 房主侧流程

房主收到 `member-joined` 事件后，执行路径选择并为该加入者创建传输：

```
TunnelManager._setupHostTransportForGuest(memberId, guestNetwork)
  │
  ├── 调用 selectPath(hostNetwork, guestNetwork) 获取最优路径
  │
  ├── 根据最优路径创建主传输：
  │   ├── relay  ──▶ 创建 RelayPeerTransport（当前行为，不变）
  │   ├── ipv6   ──▶ 创建 Ipv6DirectTransport（passive 模式）
  │   │                → connect() resolve 后获取 localPort
  │   │                → sendSignal({ type: 'ipv6-address', ip, port })
  │   └── p2p    ──▶ 创建 P2pTransport（passive 模式）
  │                    → connect() resolve 后获取 localPort
  │                    → sendSignal({ type: 'p2p-address', ip, port })
  │
  └── 如果主传输非 relay：额外创建 RelayPeerTransport 作为降级备用
       → _relayFallbacks Map 管理
       → 当 relay 数据到达时自动切换到 RelayPeerTransport
```

**降级备用机制（Relay Fallback）**：
当房主为加入者选择了 IPv6 或 P2P 传输后，额外创建一个 RelayPeerTransport 作为备用。如果加入者侧因 NAT 变化等原因无法直连而自动降级到中继，房主侧已经准备好对应的 RelayPeerTransport，数据不会丢失：

```
P2pTransport 活跃时
  └── _relayFallbacks 中同时存在 RelayPeerTransport（静默待命）
       └── 加入者侧降级到中继
            → 房主 RelayClient 收到 relay 数据
            → _relayFallbacks 检测到匹配 memberId 的数据
            → 自动切换：主传输关闭，RelayPeerTransport 接管
            → LocalTunnelClient 无需重建，传输层透明切换
```

##### 加入者侧流程

```
TunnelManager.joinRoom(roomCode)
  │
  ├── 本机网络检测
  ├── relayClient.joinRoom() → 获取房主 NetworkInfo
  │
  ├── 条件性等待房主信号（并行，带 1500ms 超时）：
  │   ├── 本机有 IPv6 → waitForIpv6Signal() 等待房主 IPv6 地址
  │   └── 本机可 P2P  → waitForP2pSignal() 等待房主 P2P 地址
  │
  ├── 调用 selectPath() + getSortedPaths() 获取有序路径列表
  │
  └── 按序尝试路径：
      paths[0] → connect() → 成功? → 使用此路径
      paths[1] → connect() → 成功? → 降级到此路径
      paths[2] → connect() → 成功? → 使用 Relay（兜底）
```

**信号等待**：
加入者不无条件等待所有信号，而是根据本机网络能力决定等待哪些信号：

| 本机网络条件 | 等待的信号 | 超时 |
|-------------|-----------|------|
| IPv6 可用 | IPv6 地址信号 | 1500ms |
| 有公网 IP（可 P2P）| P2P 地址信号 | 1500ms |
| 两者都不可用 | 不等待，直接走 Relay | — |

超时后不影响体验，按路径列表顺序降级到下一候选。

##### 生命周期管理

```
成员加入
  │
  ├── _guestTransports.set(memberId, transport) — 主传输
  ├── _relayFallbacks.set(memberId, relayTransport) — 备用传输（非 relay 时）
  │
  成员离开 / 房间关闭
  │
  ├── _cleanupGuestTransport(memberId):
  │   ├── 关闭主传输
  │   ├── 关闭 relay 备用传输
  │   └── 关闭关联的 LocalTunnelClient
  ├── leaveRoom():
  │   ├── 清理所有 _guestTransports
  │   ├── 清理所有 _relayFallbacks
  │   ├── 关闭本机 LocalTunnelClient
  │   └── 关闭 RelayClient
```

#### 5.4.10 KCP UDP 传输 (`kcp-transport.ts`)

##### 概述

`KcpTransport` 是 KCP（可靠 UDP）协议的传输层实现，用于 UDP P2P 打洞场景。在 UDP 之上叠加 KCP 协议提供可靠有序传输，支持低延迟模式（nodelay + 快速重传 + 无拥塞控制），适合实时游戏数据流。

##### 架构

```
┌──────────┐      KCP 层 (可靠传输)      ┌──────────┐
│ 发送方    │ ◄────────────────────────► │ 接收方    │
│          │   UDP datagram (基础承载)    │          │
│ KcpTransport  ◄──────────────────────►  KcpTransport │
└──────────┘                            └──────────┘
```

- **UDP 层**：`dgram.Socket` 收发原始 UDP 数据报
- **KCP 层**：在 UDP 之上实现 ARQ（自动重传请求）、流控、窗口管理
- **握手**：active 端发送空包打洞 → passive 端回复 1 字节 ACK → 双向可达后初始化 KCP 实例

##### 已知问题 — KCP 内部缓冲区复用导致包头损坏

**根因**：KCP 协议栈的 `flush()` 方法内部使用固定缓冲区 `this.buf` 打包报文段。当缓冲区满或 flush 结束时，output 回调把 `this.buf.slice()` 创建的 Buffer 视图（与 `this.buf` 共享同一底层 `ArrayBuffer`）直接传给 `udpSocket.send()`。

由于 `dgram.Socket.send()` 是异步操作，在 libuv 实际发出数据前，下一次 `flush()` 循环可能已经往 `this.buf` 写入了新数据。此时前一个还在排队发送中的 UDP 数据报内容被覆盖，导致 KCP 头部损坏：

| 损坏的字段 | 典型值 | 预期值 |
|-----------|--------|-------|
| `conv` | `0x51515151`（KCP_CMD_PUSH 重复 4 次）| `0` |
| `token` | 随机值 | `0` |
| `cmd` | 乱码 | 81 / 82 |

对端 `kcp.input()` 检查 `conv !== this.conv` 后直接返回 `-1`，丢弃所有数据包，连接完全中断。

**影响范围**：大数据量传输时触发概率接近 100%（小数据由于单次 flush 即可完成，较少触发）。

**修复**：在 output 回调中用 `Buffer.from(buf)` 创建独立副本再发送：

```typescript
const copy = Buffer.from(buf)
this._udpSocket.send(copy, 0, copy.length, ...)
```

`Buffer.from(buf)` 分配新内存并复制内容，确保 `socket.send()` 操作的数据独立于 KCP 内部缓冲区，不受后续 flush 影响。

**验证**：单元测试覆盖 3 个场景：
- 100KB 基础传输 — 验证连接可用
- 1.5MB MC 世界数据突发（53 个分片，大小范围 47B–21KB）— 验证大数据完整性
- 双向交替传输（100 条发送 + 回复）— 验证不死锁

##### 帧协议与带内控制信号

KcpTransport 在 KCP 数据流上使用 1 字节帧类型前缀，实现数据帧与控制帧的带内复用：

```
帧格式：  [1B frameType][payload...]

0x00 = DATA   — 应用数据帧，接收方剥除帧头后发射 DATA 事件
0x01 = RESET  — 控制帧，无 payload，接收方发射 RESET 事件
```

**设计动机**：解决游戏退出重连时的竞态条件。旧方案中，重置信令走 Relay Signal（WebSocket TCP），游戏数据走 KCP（UDP），UDP 比 TCP 快导致信号晚于数据到达，新旧数据混合破坏游戏协议。

**带内重置流程**：
- 加入者侧游戏客户端重连 → `KcpTransport.sendControl('reset')` → KCP 流中的 `[0x01]` 帧
- 房主侧 KCP 收到 `[0x01]` → 发射 `TRANSPORT_EVENTS.RESET` → `LocalTunnelClient._handleReset()` 销毁旧 socket
- 后续新游戏数据 `[0x00][payload]` → 缓冲 → 惰性重连 → 写入新连接

由于 KCP 保证可靠有序，RESET 帧在残留旧数据之后、新游戏数据之前到达，时序正确。

```
▶ signal (客户端 → 服务器)
  └ 功能：转发 TCP 连接信息给房间内特定成员
  {
    type: "signal",
    to: string,          // 目标 peerId
    signalData: {
      type: "connection-request" | "connection-accept" | "nat-info",
      publicIp: string,      // 公网 IPv4
      publicPort: number,    // 公网端口
      privateIp?: string,    // 私有 IP（同 LAN 检测优先）
      privatePort?: number   // 私有端口
    }
  }

◀ signal (服务器 → 客户端)
  └ 功能：转发来自其他成员的信号
  {
    type: "signal",
    from: string,
    signalData: {
      type: "connection-request" | "connection-accept" | "nat-info",
      publicIp: string,
      publicPort: number,
      privateIp?: string,
      privatePort?: number
    }
  }
```

#### 5.6.4 NetworkInfo 数据格式

```typescript
{
  ipv6: {
    available: boolean,       // 是否有全局单播 IPv6 地址
    hasPublicV6: boolean,     // 是否有可达的公网 IPv6
    addresses: string[]       // IPv6 地址列表（排除 fe80::）
  },
  ipv4: {
    natType: "none" | "full-cone" | "restricted-cone"
            | "port-restricted-cone" | "symmetric" | "unknown",
    publicIp: string,         // 公网 IPv4 地址
    publicPort: number,       // 公网端口映射
    mappingBehavior: "endpoint-independent" | "address-dependent"
                   | "address-and-port-dependent" | "unknown",
    filteringBehavior: "endpoint-independent" | "address-dependent"
                      | "address-and-port-dependent" | "unknown"
  }
}
```

#### 5.6.5 TCP 数据中继协议

当走 Relay 模式时，TCP 数据通过 WebSocket 二进制帧传输：

```typescript
// 帧格式：[4B Big-Endian payloadLen][payload bytes]
// WebSocket 发送类型为 binary（opcode 0x2）
const header = Buffer.alloc(4)
header.writeUInt32BE(payload.length, 0)
const frame = Buffer.concat([header, payload])
ws.send(frame)
```

- 房主到加入者的数据：Relay 服务器接收后通过该加入者的 WebSocket 连接转发
- 加入者到房主的数据：Relay 服务器接收后通过房主的 WebSocket 连接转发
- 接收方根据帧头 4 字节长度前缀提取负载数据
- 长度前缀为 Big-Endian UInt32，支持最大 4GB 单帧
- 控制消息（JSON 文本帧）与数据帧（二进制帧）在同一条 WebSocket 连接上共存，通过帧类型区分

## 8. 安全考虑

### 8.1 隧道安全
- Relay Server 通信使用 TLS 加密（WSS / TCP over TLS）
- P2P 连接使用 TCP 直连，数据通过 TCP Socket 传输
- 房间码作为临时身份凭证，防止未授权连接

### 8.2 应用安全
- Electron 启用上下文隔离（`contextIsolation: true`）
- 禁用 `nodeIntegration`，使用 `preload.ts` + `contextBridge`
- 渲染进程通过预定义的 IPC 通道与主进程通信
- 输入校验：房间码格式校验，端口号范围校验

### 8.3 隐私
- LAN 扫描数据仅保存在本地，不上传
- 隧道数据端到端加密（P2P 模式）或 TLS 加密（Relay 模式）
- 生产环境日志不输出公网 IPv4/IPv6 地址（NODE_ENV 条件控制，开发阶段可见）
- 用户可随时断开所有连接

## 12. 待优化方案

### 12.1 UDP + KCP P2P 打洞（计划中）

#### 12.1.1 动机

当前 P2P 使用 TCP 直连，穿透率仅覆盖约 40-50%（Full Cone NAT + 同 LAN + 公网 IP）。
Restricted Cone NAT 的 TCP 打洞成功率很低，大量用户被迫走 Relay 中继。

UDP 打洞成功率远高于 TCP（约 80-90%），因为 UDP 无连接状态，NAT 只需看到
出站包就会建立映射，入站包只要匹配映射即放行。

#### 12.1.2 实现方案

新增 `UdpTransport`，实现 `Transport` 接口，与现有传输层无缝替换：

```
┌─────────────────────────────────────────┐
│            Transport 接口                 │
│  connect / send / disconnect / events    │
├─────────────────────────────────────────┤
│          UdpTransport (新增)              │
│  ┌─────────────────────────────────────┐ │
│  │  UDP 打洞阶段                        │ │
│  │  1. Passive 绑定本地 UDP 端口        │ │
│  │  2. 双方通过 STUN 获取公网映射       │ │
│  │  3. 双方互发 UDP 包建立 NAT 映射     │ │
│  │  4. 双向通信验证                     │ │
│  ├─────────────────────────────────────┤ │
│  │  数据传输阶段                        │ │
│  │  UDP + KCP 可靠传输                  │ │
│  │  - 自定义 FEC 前向纠错               │ │
│  │  - 可配置 nodelay 低延迟模式          │ │
│  │  - 自动重传 + 有序交付               │ │
│  │  - 拥塞控制（适合游戏小包场景）       │ │
│  └─────────────────────────────────────┘ │
└─────────────────────────────────────────┘
```

KCP 优势（对比原生 TCP）：

| 场景 | TCP | UDP + KCP |
|------|-----|-----------|
| 丢包重传 | 等待 RTO（至少 200ms） | 可配置 10-30ms 快速重传 |
| 拥塞控制 | 复杂（CUBIC 等） | 可关闭，适合 LAN 级 P2P |
| 头开销 | 20 字节 | 24 字节（KCP） |
| 流控 | 内核自动 | 可自定义 |
| FEC | 不支持 | 支持前向纠错 |

#### 12.1.3 路径选择器更新

```typescript
// 当前                            → 后续
[ipv6, p2p(tcp), relay]           [ipv6, udp, p2p(tcp), relay]
                                   // UDP 优先级高于 TCP 直连
                                   // 因为穿透率更高
```

#### 12.1.4 实施时间

建议在 Phase 4（UI）+ Phase 5（测试优化）完成后实施。
现有 `Transport` 接口完全兼容，新增 `UdpTransport` 类即可，不影响已有代码。
依赖：`kcp` npm 包或基于 `dgram` 自行实现。
