# 局域网游戏联机工具 — 技术文档

## 1. 项目概述

### 1.1 目标
开发一款桌面应用，帮助非技术用户将**单机局域网联机游戏**（如 Minecraft、Terraria 等）通过互联网进行联机。用户无需了解端口转发、路由器配置、防火墙等网络知识，即可一键创建或加入远程游戏房间。

### 1.2 核心场景
- **房主（Host）**：启动游戏后打开本软件，软件自动检测到本机运行的游戏服务器 → 一键创建房间 → 生成房间码/链接分享给好友
- **加入者（Join）**：输入好友的房间码 → 软件自动建立隧道连接 → 加入者启动游戏 → 在局域网列表中看到房主的游戏 → 点击加入

### 1.3 支持平台
| 平台 | 支持等级 |
|------|---------|
| Windows 10/11 | 原生支持 |
| macOS 11+ (Intel / Apple Silicon) | 原生支持 |
| Linux (Ubuntu 20.04+, 主要发行版) | 原生支持 |

```
│  ┌──────┴──────────────────────────────────────────┐  │
│  │            IPC 桥接层 (ipc-handlers.js)          │  │

│  │  └──────────┘ └──────────┘ └────────────────┘   │  │

│  ┌─────────────────────────────────────────────────┐  │

│  └─────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────┘
```

**关键决策：核心引擎运行在主进程**。原因：
1. 核心引擎需要访问 Node.js 原生网络 API（`dgram`、`net`），这些在渲染进程中不可用（即使启用 `nodeIntegration` 也不推荐）
2. 渲染进程崩溃不应中断正在进行的隧道连接
3. 系统托盘可以在窗口关闭后继续运行

### 4.2 三层架构

```
┌─────────────────────────────────────────────┐
│         表现层 (Presentation)                 │
│  Vue 3 组件 + Pinia 状态管理                 │
│  关注点：UI 渲染、用户交互、本地化            │
├─────────────────────────────────────────────┤
│         桥接层 (Bridge)                       │
│  Electron IPC (contextBridge + ipcHandlers)  │
│  关注点：安全地暴露核心 API 给渲染进程         │
├─────────────────────────────────────────────┤
│         核心层 (Core)                         │
│  TypeScript 引擎 (平台无关)                   │
│  关注点：网络发现、隧道建立、P2P 通信          │
└─────────────────────────────────────────────┘
```

### 4.3 数据流

```
【房主创建房间】
  1. Renderer: 点击"创建房间"
  2. IPC:      create-room
  3. Core:     GameDetect 查找本机游戏进程 → Discovery 确认端口
  4. Core:     NetworkDetect 检测本机网络（IPv6 能力、NAT 类型）
  5. Core:     Tunnel 建立本地隧道服务端（监听 IPv4 + IPv6）
  6. Core:     RelayClient 上报房间信息 + 房主网络信息到 Relay Server
  7. IPC:      返回 房间码 + 房主网络状态
  8. Renderer: 显示房间码 + 网络状态 + 连接状态

【加入者加入房间】
  1. Renderer: 输入房间码 → 点击"加入"
  2. IPC:      join-room
  3. Core:     NetworkDetect 检测本机网络（IPv6 能力、NAT 类型）
  4. Core:     RelayClient 从 Relay Server 获取房主网络信息
  5. Core:     ConnectionManager 执行路径选择：
                ┌─ 双方都有 IPv6 ──▶ IPv6 TCP 直连
                ├─ 仅 IPv4 + 可 P2P ──▶ WebRTC P2P
                └─ 仅 IPv4 + 无法 P2P ──▶ Relay 中继
  6. Core:     本地监听 127.0.0.1:虚拟端口
  7. IPC:      返回 虚拟端口号 + 连接方式
  8. Renderer: 提示"已通过 XXX 方式连接，请连接 127.0.0.1:端口"
  9. User:     在游戏中添加局域网游戏 → 输入 127.0.0.1:端口
```

## 5. 核心功能设计

### 5.1 LAN 游戏发现 (`src/core/discovery/`)

#### 5.1.1 UDP 广播扫描

```
Host 机器：
  → 广播 UDP 包到 255.255.255.255:发现端口
  → 包含本机信息（游戏名称、端口、版本）
  → 等待响应

Scanner 机器：
  → 监听 UDP 发现端口
  → 收到广播后解析游戏信息
  → 通过 IPC 通知渲染进程更新列表
```

- 扫描端口范围：可配置，默认扫描常见游戏端口
- 扫描间隔：每 3 秒广播一次，持续 30 秒
- 游戏退出后自动从列表移除（超时 15 秒无响应）

#### 5.1.2 已知游戏数据库 (`game-db.ts`)

```typescript
interface GameEntry {
  id: string;           // "minecraft-java"
  name: string;         // "Minecraft Java Edition"
  protocol: 'tcp' | 'udp';
  defaultPort: number;
  detectPorts: number[]; // 常见端口
  processNames: string[]; // 进程名（用于本地检测）
  ping?: (host: string, port: number) => Promise<GameInfo>;
}
```

内置支持的游戏（持续扩充）：

### 5.2 网络检测 (`src/core/network-detect/`)

#### 5.2.1 概述

每次加入者连接房间前，双方独立进行网络环境检测。房主在网络检测后将其结果上传到 Relay Server，加入者获取房主信息后与本机结果对比，选择最优连接路径。

#### 5.2.2 检测内容

```typescript
interface NetworkInfo {
  ipv6: {
    available: boolean           // 是否有全局单播 IPv6 地址
    hasPublicV6: boolean         // 是否具有可达的公网 IPv6
    addresses: string[]          // 本机 IPv6 地址列表（排除链路本地 fe80::）
  }
  ipv4: {
    natType: NatType             // NAT 类型检测结果
    publicIp: string             // 公网 IPv4 地址（通过 STUN 获取）
    publicPort: number           // 公网端口映射
  }
}

enum NatType {
  None = 'none',                 // 无 NAT（公网 IP）
  FullCone = 'full-cone',       // 全锥型 NAT
  RestrictedCone = 'restricted-cone',  // 限制锥型 NAT
  PortRestrictedCone = 'port-restricted-cone',  // 端口限制锥型 NAT
  Symmetric = 'symmetric',       // 对称型 NAT ❌ 无法 P2P
  Unknown = 'unknown',
}
```

#### 5.2.3 检测流程

```
NetworkDetect.run():
  │
  ├── IPv6 检测（并行）───────▶ net.createServer() 监听 IPv6 端口
  │                            net.createConnection() 尝试连接公网 IPv6 可达性
  │                            扫描本机网卡获取 IPv6 地址列表
  │
  └── NAT 检测（并行）────────▶ 通过 STUN 服务器 (stun.l.google.com:19302)
                               RFC 3489 流程判断 NAT 类型
                               获取公网 IP 和端口

  返回 NetworkInfo
```

- NAT 检测使用 RFC 3489 经典 STUN 协议（非 RFC 5389），因为完整的 RFC 3489 实现可以区分 NAT 子类型
- IPv6 检测使用操作系统网络接口 API + 实际 TCP 监听验证
- 两项检测并行执行，总耗时约 1-2 秒
- 检测结果缓存在 Room 生命周期内，不重复检测

### 5.3 连接路径选择 (`src/core/connection/`)

#### 5.3.1 路径选择算法

```
PathSelector.select(hostInfo: NetworkInfo, guestInfo: NetworkInfo): ConnectionPath

  优先级 1 ── IPv6 直连
    条件：hostInfo.ipv6.hasPublicV6 === true
          guestInfo.ipv6.hasPublicV6 === true
    方式：TCP Socket 直连（net.createConnection）
    优点：零 NAT 问题、无中继成本、最低延迟、最高吞吐
    └── 失败 ──▶ 降级到优先级 2

  优先级 2 ── IPv4 P2P
    条件：hostInfo.ipv4.natType !== Symmetric
          guestInfo.ipv4.natType !== Symmetric
    方式：WebRTC DataChannel + STUN
    优点：无中继成本、延迟低
    └── 失败 ──▶ 降级到优先级 3

  优先级 3 ── Relay 中继
    条件：无条件（兜底方案）
    方式：双方连接到 Relay Server，双向转发
    优点：100% 可用（依赖服务器）
```

#### 5.3.2 每加入者独立选路

```
           ┌─────────── 加入者 A (有 IPv6) ───▶ IPv6 TCP 直连
           │
房主 ──────┼─────────── 加入者 B (无 IPv6, FullCone NAT) ───▶ IPv4 P2P
           │
           └─────────── 加入者 C (无 IPv6, Symmetric NAT) ───▶ Relay 中继

每个加入者独立走自己的最优路径，互不影响。
```

- 房主持有所有已建立连接的引用，流量分别转发到对应的游戏端口
- 加入者 A 的高吞吐 IPv6 直连不影响加入者 C 的中继连接
- 中继服务器仅服务无法 P2P 的加入者，大幅降低服务器负载

#### 5.3.3 动态切换

如果某一加入者的连接方式中断，自动降级到下一优先级：

```
IPv6 直连断开 ──▶ 尝试 P2P ──▶ 失败则走 Relay（保持连接不断）
```

切换过程对上层游戏透明（游戏 TCP 连接无需重连，新隧道连接后继续转发）。

### 5.4 TCP 隧道 (`src/core/tunnel/`)

#### 5.4.1 隧道架构

```
IPv6 直连模式：
┌─────────┐      ┌──────────┐          TCP          ┌──────────┐      ┌─────────┐
│ 游戏服   │◄────►│ 本地隧道  │◄══════════════════════►│ 本地隧道  │◄────►│ 游戏客户 │
│ :25565   │ TCP  │ (房主端)  │  v6 TCP 直接互联       │ (加入者)  │ TCP  │ :25565   │
└─────────┘      └──────────┘                        └──────────┘      └─────────┘

P2P 模式：
┌─────────┐      ┌──────────┐   WebRTC DataChannel   ┌──────────┐      ┌─────────┐
│ 游戏服   │◄────►│ 本地隧道  │◄══════════════════════►│ 本地隧道  │◄────►│ 游戏客户 │
│ :25565   │ TCP  │ + P2P桥   │   STUN 穿透 P2P       │ + P2P桥   │ TCP  │ :25565   │
└─────────┘      └──────────┘                        └──────────┘      └─────────┘

Relay 模式：
┌─────────┐      ┌──────────┐      ┌──────────────┐      ┌──────────┐      ┌─────────┐
│ 游戏服   │◄────►│ 本地隧道  │◄────►│  中继服务器    │◄────►│ 本地隧道  │◄────►│ 游戏客户 │
│ :25565   │ TCP  │ (房主端)  │ TLS  │ relay.example │ TLS  │ (加入者)  │ TCP  │ :25565   │
└─────────┘      └──────────┘      └──────────────┘      └──────────┘      └─────────┘
```

#### 5.4.2 连接策略总览

| 优先级 | 连接方式 | 条件 | 优势 | 劣势 |
|--------|---------|------|------|------|
| 1 (最优) | IPv6 TCP 直连 | 双方均有全局 IPv6 | 原生 TCP、最低延迟、零中继成本 | IPv6 尚未全面普及 |
| 2 | IPv4 P2P (WebRTC) | 双方 NAT 均可穿透 | 低延迟、无中继成本 | 对称 NAT 无法穿透 |
| 3 (兜底) | Relay 中继 | 无条件 | 100% 可达 | 依赖服务器、延迟增加 |

### 5.5 P2P 直连 (`src/core/p2p/`)

#### 5.5.1 WebRTC 方案

使用 `node-datachannel` 在 Node.js 中实现 WebRTC：

- **信令**：通过 Relay Server 的 WebSocket 交换 SDP / ICE Candidate
- **数据通道**：建立 P2P 数据通道后，将 TCP 数据流打包通过数据通道传输
- **STUN**：内置 Google STUN 服务器列表（`stun.l.google.com:19302` 等）
- 仅当双方 IPv6 检测失败且 NAT 类型非 Symmetric 时才启用

#### 5.5.2 TCP ↔ DataChannel 桥接 (`relay-peer.ts`)

```
本地 TCP Socket ──► 数据打包 ──► WebRTC DataChannel ──► 远端 DataChannel ──► TCP 解包 ──► 本地游戏
```

### 5.7 中继服务器协议规范

中继服务器为独立项目（仓库 `simple-game-relay`），**当前阶段只定义协议，不实现服务器代码**。客户端 Relay Client 按此协议实现，后续再用 Node.js 实现服务端。

#### 5.7.1 传输层

| 通道 | 协议 | 端口 | 用途 |
|------|------|------|------|
| 信令通道 | WebSocket (WSS) | 443 | JSON 消息：房间管理、NetworkInfo 交换、ICE 信令 |
| 数据通道 | WebSocket (WSS) | 443 | TCP 数据中继（仅兜底模式） |

- 生产环境使用 WSS（TLS），开发环境可使用 WS
- 信令通道和数据通道共用同一个 WebSocket 连接，通过消息类型区分

#### 5.7.2 消息格式

所有消息为 JSON 格式，顶层结构：

```typescript
// 客户端 → 服务器
{
  type: string,       // 消息类型
  requestId?: string, // 请求 ID（可选，用于请求-应答匹配）
  ...                 // 消息体
}

// 服务器 → 客户端
{
  type: string,
  requestId?: string, // 回显请求 ID
  success: boolean,
  error?: string,     // 仅 success === false 时存在
  ...                 // 消息体
}
```

#### 5.7.3 消息类型定义

##### 房间管理

```
▶ create-room (客户端 → 服务器)
  └ 功能：房主创建房间，上传本机网络检测结果
  └ 触发时机：房主点击"创建房间"、完成网络检测后
  {
    type: "create-room",

    gamePort: number,         // 25565
    gameName: string,         // "Minecraft Java Edition"

    version: string           // 客户端版本 "1.0.0"
  }

◀ room-created (服务器 → 客户端)
  └ 功能：返回房间码
  {
    type: "room-created",
    requestId: string,
    success: true,

  }

◀ room-created (失败)
  {
    type: "room-created",
    requestId: string,
    success: false,

  }


▶ join-room (客户端 → 服务器)
  └ 功能：加入者加入房间，上传本机网络检测结果
  └ 触发时机：加入者输入房间码、完成网络检测后
  {
    type: "join-room",

  }

◀ room-joined (服务器 → 客户端)
  └ 加入者收到：
  {
    type: "room-joined",

    gameId: string,
    gameName: string,

  }
  └ 房主收到推送（新成员加入通知）：
  {
    type: "peer-joined",

    peerNetworkInfo: NetworkInfo,  // 新加入者的网络检测结果
    memberCount: number
  }


▶ leave-room (客户端 → 服务器)
  └ 功能：主动离开房间
  {
    type: "leave-room",

  }

◀ peer-left (服务器 → 客户端)
  └ 功能：通知房间内其他成员有人离开
  {

  }


▶ heartbeat (客户端 → 服务器)
  └ 频率：每 10 秒
  {
    type: "heartbeat",

  }

◀ heartbeat-ack (服务器 → 客户端)
  {
```

##### P2P 信令中继

当路径选择器判定双方走 P2P 时，通过中继服务器交换 WebRTC 连接信息：

```
▶ signal (客户端 → 服务器)
  └ 功能：转发 SDP Offer/Answer 或 ICE Candidate 给房间内特定成员
  {
    type: "signal",
    to: string,          // 目标 peerId
    data: {
      type: "offer" | "answer" | "ice-candidate",
      sdp?: string,      // SDP 描述 (offer/answer)
      candidate?: string // ICE Candidate (ice-candidate)
    }
  }

◀ signal (服务器 → 客户端)
  └ 功能：转发来自其他成员的信号
  {
    type: "signal",
    from: string,
    data: {
      type: "offer" | "answer" | "ice-candidate",
      sdp?: string,
      candidate?: string
    }
  }
```

#### 5.7.4 NetworkInfo 数据格式

```typescript
{
  ipv6: {
    available: boolean,       // 是否有全局单播 IPv6 地址
    hasPublicV6: boolean,     // 是否有可达的公网 IPv6
    addresses: string[]       // IPv6 地址列表（排除 fe80::）
  },
  ipv4: {
    natType: "none" | "full-cone" | "restricted-cone"
            | "port-restricted-cone" | "symmetric" | "unknown",
    publicIp: string,         // 公网 IPv4 地址
    publicPort: number        // 公网端口映射
  }
}
```

#### 5.7.5 TCP 数据中继协议

当走 Relay 模式时，TCP 数据通过 WebSocket 按帧传输：

```typescript
// 每个 WebSocket 消息为一个数据帧
{
  type: "relay-data",
  from: string,          // 发送者 peerId
  sequence: number,      // 序列号（排序和丢包检测）
  data: string,          // Base64 编码的 TCP 数据
  ack: number            // 已确认的最后一个序列号（流控）
}
```

- 房主到加入者的数据：`from` 为房主 ID，加入者收到后写入本地 TCP 游戏端口
- 加入者到房主的数据：`from` 为加入者 ID，房主收到后转发给对应游戏端口
- 序列号用于检测丢包和去重；`ack` 用于简单流控

##### 统一数据帧

无论 Relay 还是 P2P 模式，数据帧在传输层之上保持统一格式，便于路径切换：

```typescript
{
  frameType: "game-data",   // 数据帧类型
  sequence: number,          // 序列号
  payload: Buffer,           // 变长 TCP 数据
  timestamp: number          // 时间戳（延迟统计）
}
```

## 8. 安全考虑

### 8.1 隧道安全
- Relay Server 通信使用 TLS 加密（WSS / TCP over TLS）
- P2P 连接使用 DTLS-SRTP（WebRTC 内置加密）
- 房间码作为临时身份凭证，防止未授权连接

### 8.2 应用安全
- Electron 启用上下文隔离（`contextIsolation: true`）
- 禁用 `nodeIntegration`，使用 `preload.ts` + `contextBridge`
- 渲染进程通过预定义的 IPC 通道与主进程通信
- 输入校验：房间码格式校验，端口号范围校验

### 8.3 隐私
- LAN 扫描数据仅保存在本地，不上传
- 隧道数据端到端加密（P2P 模式）或 TLS 加密（Relay 模式）
- 用户可随时断开所有连接

## 10. 自动更新机制

### 10.1 更新源策略

```
启动时检查更新 ──┬──▶ Gitee API ──▶ 成功? ──▶ 使用 Gitee 版本信息
                │                └── 失败 ──▶ GitHub API ──▶ 成功? ──▶ 使用 GitHub 版本信息
                │                                          └── 失败 ──▶ 跳过本次检查
                └── 缓存结果 (1 小时 TTL)
```

**策略**：
- **Gitee 优先**：国内用户访问 Gitee 速度快且稳定
- **GitHub 降级**：Gitee 不可用时自动切换到 GitHub Releases
- **静默检查**：启动后延迟 3 秒异步检查，不阻塞 UI
- **缓存**：1 小时内不重复请求，避免触发 API 频率限制

```
│  v1.0.1 · ✓ 已是最新版本                    │  ← 最新版
│                                            │
└────────────────────────────────────────────┘

有更新时的底部状态（三种情况）：
┌──────────────────────────────────────┐
│ v1.0.0 · 正在检查更新...             │  ← 检查中（启动后显示 3 秒）
└──────────────────────────────────────┘

┌──────────────────────────────────────┐
│ v1.0.0 · ✓ 已是最新版本              │  ← 无需更新（默认态）
└──────────────────────────────────────┘

┌──────────────────────────────────────┐
│ v1.0.0 · 🔄 新版本 v1.0.1 立即更新   │  ← 有更新（点击触发下载）
└──────────────────────────────────────┘
```

- 更新日志从 Gitee/GitHub Releases 的 body 字段获取（Markdown 渲染为列表）
- 版本状态固定在首页最底部，不影响操作按钮布局
- 有更新时底部显示"立即更新"按钮，点击后开始下载安装

### 10.5 安装包匹配规则

根据当前操作系统自动选择对应架构的安装包：

| 平台 | 匹配规则 |
|------|---------|
| Windows | `*-win.exe` 或 `*-Setup-*.exe` |
| macOS (Intel) | `*-mac.dmg` 或 `*-x64.dmg` |
| macOS (Apple Silicon) | `*-arm64.dmg` 或 `*-mac-arm64.dmg` |
| Linux (deb) | `*_amd64.deb` |
| Linux (AppImage) | `*.AppImage` |

### 10.6 遇到的挑战与应对

| 挑战 | 应对方案 |
|------|---------|
| Gitee API 频率限制 (60 req/h 未认证) | 1 小时本地缓存 + 可选的 User Token 配置 |
| Gitee 下载限速 | 支持断点续传、多线程下载 |
| GitHub 被 GFW 干扰 | 作为纯降级方案，且支持 Token 认证 |
| 安装包校验 | 下载后校验 SHA256 + 代码签名 |
| 静默更新失败 | 提供手动下载链接，引导用户去 Releases 页面 |
